## Part 0. 기술 셋업

> 코드 짜기 **전에** 손으로 미리 준비해야 하는 환경 설정.

### 0.1 아키텍처 개요

- **인증**: Supabase Auth (이메일+비밀번호, Google OAuth)
- **DB**: Supabase Postgres
- **메일 발송**: Supabase 내장 (비밀번호 재설정 메일)
- **클라이언트/서버 분리**: `@supabase/ssr` 사용
  - 브라우저 코드 → anon key (공개 가능)
  - 서버 코드(Server Components, Route Handlers, Server Actions) → anon key + 쿠키 기반 세션
  - 관리자 작업(필요 시) → service_role key (서버 only, 절대 클라이언트 노출 금지)
- **인증 보호**: Next.js middleware로 비로그인 사용자를 `/login`으로 리다이렉트
- **보안**: Supabase Row Level Security(RLS)로 "내 데이터는 나만 본다" 강제

### 0.2 의존성 (설치할 패키지)

```bash
npm install @supabase/supabase-js @supabase/ssr
```

> **패키지 매니저: npm으로 통일.** (`pnpm-lock.yaml`은 삭제됨)

### 0.3 환경변수

`.env.local` (로컬 개발용, **Git에 올리지 말 것**):

```
NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=xxxxx
SUPABASE_SERVICE_ROLE_KEY=xxxxx
```

- `NEXT_PUBLIC_*` → 브라우저에서 사용 가능 (공개돼도 안전)
- `SUPABASE_SERVICE_ROLE_KEY` → **서버 코드에서만** 사용. 누출 시 누구나 모든 데이터 접근 가능.
- Vercel 대시보드에도 동일한 키를 Production/Preview 환경에 등록 필요. (`vercel env pull`로 로컬과 동기화)

### 0.4 외부 설정 (코드 외 작업)

#### A) Supabase 프로젝트 생성
- Vercel Marketplace → Supabase Integration 설치 → 새 프로젝트 생성
- 생성 후 위 환경변수 3개가 Vercel에 자동 주입됨
- **Region**: `Northeast Asia (Seoul) ap-northeast-2` ← 한국 사용자 응답 최적화

#### B) Google Cloud Console — OAuth 클라이언트 발급
1. https://console.cloud.google.com → 새 프로젝트 생성 (예: `investment-dashboard`)
2. "OAuth 동의 화면" 설정 (앱 이름, 지원 이메일)
3. "사용자 인증 정보 → OAuth 클라이언트 ID 만들기" → 유형: 웹 애플리케이션
4. **승인된 리디렉션 URI** 등록:
   - `https://<supabase-project-ref>.supabase.co/auth/v1/callback`
5. 발급된 **클라이언트 ID**, **클라이언트 보안 비밀** 메모

#### C) Supabase Dashboard — OAuth & URL 설정
1. Authentication → Providers → Google 활성화 → 위 클라이언트 ID/Secret 입력
2. Authentication → URL Configuration
   - **Site URL**: `http://localhost:3000` (개발) / `https://investment-dashboard-sjv4.vercel.app` (배포)
   - **Redirect URLs**:
     - `http://localhost:3000/auth/callback`
     - `https://investment-dashboard-sjv4.vercel.app/auth/callback`

## Part 1. 화면 설계서

### 화면 목록
- `/signup` — 회원가입
- `/login` — 로그인
- `/reset-password` — 비밀번호 재설정 요청
- `/reset-password/new` — 새 비밀번호 입력 (메일 링크로 진입)
- `/mypage` — 마이페이지 (닉네임/자기소개 관리)
- 대시보드 `/` — 기존 화면. 인증·닉네임 추가에 따른 변경사항만 명시.

각 화면은 아래 9가지 항목을 모두 채운다.

---

### 화면 1: `/signup` (회원가입)

#### 1) 화면 정보
- 화면명: 회원가입
- 목적: 새 사용자가 이메일+비밀번호 또는 구글로 계정을 만든다
- 접근 가능 사용자: 비로그인 사용자만 (이미 로그인 상태로 진입 시 `/`로 리다이렉트)

#### 2) 컴포넌트 상세 정의
| 컴포넌트 | 타입 | 필수 | 최대 길이 | Placeholder |
|---|---|---|---|---|
| 이메일 입력 | `input[type=email]` | ✅ | 254자 | `you@example.com` |
| 비밀번호 입력 | `input[type=password]` | ✅ | 72자 | `비밀번호 (8자 이상)` |
| "보기" 토글 (비밀번호) | 아이콘 버튼 | - | - | 👁 / 🙈 아이콘 (클릭 시 type=text ↔ password) |
| 가입하기 버튼 | `button[type=submit]` | - | - | label: "가입하기" |
| "구글로 계속하기" 버튼 | `button` | - | - | 구글 로고 + "구글로 계속하기" |
| 로그인 페이지 링크 | `a` | - | - | "이미 계정이 있으신가요? 로그인" |

> 표시 이름(예: "Good morning, ___")은 가입 후 이메일 앞부분(`@` 기준 왼쪽)에서 자동으로 가져온다. 별도 입력 필드 없음.

#### 3) 유효성 검증
- **이메일**: 비어있지 않음 + 유효한 이메일 형식 (브라우저 기본 `type=email` 검사 + 서버 측 재검사)
- **비밀번호**: 최소 8자 (복잡도 강제 없음 — 대문자/숫자/특수문자 불필요), 최대 72자 (bcrypt 입력 한계)
- **검증 시점**: 입력 중 실시간
  - 비밀번호 입력란 아래에 현재 글자 수 / 최소 길이 상태를 함께 표시 (예: 회색 "8자 이상 입력" → 8자 도달 시 초록 "✓ 사용 가능")
  - 가입 버튼은 두 필드가 모두 유효해질 때까지 비활성화(disabled)

#### 4) 화면 레이아웃
- **배경**: 기존 대시보드와 동일한 다크 배경 (별도 헤더/푸터 없음)
- **본문**: 화면 정중앙에 카드 1장 (shadcn `Card`)
- **카드 내부 (위 → 아래)**:
  1. 로고/타이틀: "투자 대시보드"
  2. 화면 제목: "계정 만들기"
  3. 이메일 입력
  4. 비밀번호 입력 + 보기 토글
  5. 비밀번호 실시간 도움말 ("8자 이상 입력" / "✓ 사용 가능")
  6. 가입하기 버튼 (메인, 가로 100%)
  7. 구분선: "─── 또는 ───"
  8. 구글로 계속하기 버튼 (보조, 가로 100%, 구글 로고 포함)
  9. 푸터 링크: "이미 계정 있으신가요? **로그인**" → `/login`
- **반응형**: 카드 최대 너비 ~400px, 모바일에서는 화면 패딩만 두고 자연스럽게 줄어듦

#### 5) 사용자 액션
- 이메일/비밀번호 입력 후 "가입하기" 클릭 → 가입 처리
- "구글로 계속하기" 클릭 → 구글 OAuth 흐름
- "로그인" 링크 클릭 → `/login`으로 이동

#### 6) 에러 메시지
**필드별 (입력란 바로 아래, 빨간 글씨)**
| 케이스 | 문구 |
|---|---|
| 이메일 형식 틀림 | `이메일 형식이 올바르지 않습니다` |
| 비밀번호 8자 미달 | (별도 에러 없음 — 도움말 "8자 이상 입력"으로 처리) |

**전체 (폼 상단의 빨간 박스)**
| 케이스 | 문구 |
|---|---|
| 이미 가입된 이메일 | `이미 가입된 이메일입니다.` + 우측에 `로그인하기` 링크 (→ `/login`) |
| 네트워크/서버 오류 | `잠시 후 다시 시도해주세요.` |
| 구글 OAuth 취소/실패 | `구글 로그인을 완료하지 못했습니다. 다시 시도해주세요.` |

#### 7) UI 컴포넌트 목록
- Input × 2 (이메일, 비밀번호)
- Button × 2 (가입하기, 구글로 계속하기)
- IconButton × 1 (비밀번호 보기 토글)
- Link × 1 (로그인 페이지로)

#### 8) 상태(State)
| 상태 | 화면 표현 |
|---|---|
| **idle** | 기본. 입력 가능. 가입 버튼은 비밀번호 8자 미달이면 회색 비활성. |
| **submitting** | 가입 버튼이 `가입 중...` + 스피너로 바뀜. 입력란·구글 버튼·링크 모두 비활성. |
| **success** | 별도 UI 없이 즉시 `/` (대시보드)로 이동. |
| **error** | 폼 상단에 빨간 박스로 에러 표시 → idle 상태로 복귀 (입력 다시 가능). |

#### 9) 성공 결과
- 가입 성공 시 자동 로그인 처리 → `/` (대시보드)로 이동

### 화면 3: `/reset-password` (비밀번호 재설정 요청)

#### 1) 화면 정보
- 화면명: 비밀번호 재설정 요청
- 목적: 비밀번호를 잊은 사용자가 자신의 이메일로 재설정 링크를 받는다
- 접근 가능 사용자: 누구나

#### 2) 컴포넌트 상세 정의
| 컴포넌트 | 타입 | 필수 | 최대 길이 | Placeholder |
|---|---|---|---|---|
| 이메일 입력 | `input[type=email]` | ✅ | 254자 | `you@example.com` |
| "재설정 메일 보내기" 버튼 | `button[type=submit]` | - | - | label: "재설정 메일 보내기" |
| 로그인 복귀 링크 | `a` | - | - | "← 로그인으로 돌아가기" → `/login` |

**제출 후(sent 상태) 카드 내용 교체용:**
| 컴포넌트 | 내용 |
|---|---|
| 제목 | "메일을 확인해주세요" |
| 본문 | "[입력한 이메일]로 비밀번호 재설정 링크를 보냈습니다. 메일이 보이지 않으면 스팸함도 확인해주세요." |
| 버튼 | "로그인 화면으로 돌아가기" → `/login` |

#### 3) 유효성 검증
- **이메일**: 비어있지 않음 + 유효한 이메일 형식 (`type=email` 기본 검사 + 서버 측 재검사)
- 검증 시점: 이메일 포커스 빠질 때 + 버튼 클릭 시

## Part 2. 데이터 설계서

### 2.1 방침
Supabase가 자동으로 만들어주는 `auth.users` 시스템 테이블 + 우리만의 보조 테이블 `public.profiles` 한 개를 사용한다. profiles는 표시 이름처럼 앱이 직접 관리할 사용자 정보를 담는 짝꿍 테이블이다. (이메일 같이 `auth.users`가 이미 관리하는 정보는 중복 저장하지 않는다 — 싱크 문제 방지)

### 2.2 Supabase가 자동 관리하는 테이블 — `auth.users`
| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | uuid | 사용자 고유 ID. 미래에 만들 사용자별 테이블의 FK 대상이 됨. |
| `email` | text | 가입 이메일 (로그인 ID 역할) |
| `encrypted_password` | text | bcrypt 해시. 코드에서 직접 읽지 않음. |
| `email_confirmed_at` | timestamp | 인증 일시. 이번 스코프에선 가입 즉시 자동 채움. |
| `raw_app_meta_data` | jsonb | provider 정보 (`email` / `google`) |
| `created_at` / `updated_at` | timestamp | 가입/수정 일시 |

### 2.3 우리 코드가 다루는 데이터
**읽음**
- `id` — 세션 식별, 미래 데이터의 FK
- `email` — 표시 이름(이메일 앞부분) 도출, 인사말 표시
- `created_at` — (필요 시) 가입 일자 노출

**손대지 않음**
- `encrypted_password` — Supabase가 해시 비교를 알아서 함
- 비밀번호 재설정 토큰 — Supabase가 메일 발송과 검증을 알아서 함

### 2.4 표시 이름 처리
- 표시 이름은 `profiles.nickname`을 우선 사용
- `nickname`이 `null`이면 → `user.email.split('@')[0]` 자동 계산값 사용
- 예: `nickname = null`, email = `rogio0425@gmail.com` → 화면 표시 `rogio0425`
- 사용자가 마이페이지에서 `nickname`을 설정하면 그 값이 우선 (이후 모든 화면의 인사말·표기에 반영)

### 2.5 새로 만드는 테이블 — `public.profiles`

| 컬럼 | 타입 | 제약 | 설명 |
|---|---|---|---|
| `id` | uuid | PK, FK → `auth.users(id)` ON DELETE CASCADE | `auth.users`와 1:1 매핑 |
| `nickname` | text | nullable, **UNIQUE** | null이면 이메일 앞부분 자동 사용. 한 번 설정되면 비울 수 없음(앱 로직). |
| `bio` | text | nullable | 자기소개. 비울 수 있음. |
| `created_at` | timestamptz | NOT NULL, default `now()` | 생성 시각 |
| `updated_at` | timestamptz | NOT NULL, default `now()` | 마지막 수정 시각 |

> ⚠️ `nickname`은 DB 차원의 UNIQUE 제약을 가진다. PostgreSQL은 `NULL`을 고유하다고 보지 않으므로 여러 사용자가 NULL을 가질 수 있다(아직 닉네임 미설정 사용자가 다수일 수 있음). 빈 문자열은 저장 불가 — 앱이 trim 후 길이 0이면 거부.

**관련 트리거**
- `on_auth_user_created` (auth schema): `auth.users` INSERT 시 → `public.profiles`에 같은 id로 자동 INSERT (`nickname=null, bio=null`)
- `profiles_updated_at`: `public.profiles` UPDATE 시 → `updated_at = now()` 자동 갱신

### 2.6 RLS (Row Level Security) 정책 — `public.profiles`
- RLS **활성화**
- **SELECT**: 본인 row만 (`id = (select auth.uid())`)
- **UPDATE**: 본인 row만 (`id = (select auth.uid())`)
- **INSERT**: 정책 없음 (트리거로만 생성됨, 클라이언트 직접 INSERT 불가)
- **DELETE**: 정책 없음 (`auth.users` 삭제 시 CASCADE로 자동 삭제)

### 2.7 마이그레이션 SQL (이번 변경분)

```sql
-- 기존 display_name → nickname으로 이름 변경
alter table public.profiles rename column display_name to nickname;

-- nickname에 UNIQUE 제약 추가 (NULL은 중복 허용)
alter table public.profiles add constraint profiles_nickname_unique unique (nickname);

-- bio 컬럼 추가
alter table public.profiles add column bio text;
```

## Part 3. 동작 흐름

> 각 사용자 액션이 브라우저 → Supabase → DB → 결과로 어떻게 흐르는지 단계별로 정리. 각 흐름은 (1) 다이어그램 (2) 단계별 표 (3) 실패 분기 순으로 적는다.

---

### 3.1 흐름 1 — 이메일로 회원가입

**시나리오**: `/signup`에서 이메일·비밀번호 입력 후 "가입하기" 클릭

```
[사용자]            [브라우저 (Next.js Client)]        [Supabase Auth]            [Supabase DB]
   │ 가입하기 클릭        │                                    │                          │
   ├───────────────────►│ 1. 클라이언트 유효성 검증              │                          │
   │                    │ 2. 상태 → submitting                │                          │
   │                    │ 3. supabase.auth.signUp({          │                          │
   │                    │      email, password })             │                          │
   │                    ├───────────────────────────────────►│                          │
   │                    │                                    │ 4. bcrypt 해시            │
   │                    │                                    │ 5. INSERT auth.users    ├─►│
   │                    │                                    │ 6. 세션 토큰 발급         │
   │                    │◄───────────────────────────────────┤ 7. 응답 (user + session) │
   │                    │ 8. 세션 쿠키 자동 저장 (@supabase/ssr) │                          │
   │ 대시보드 표시         │ 9. router.push('/')                │                          │
   │◄────────────────── │                                    │                          │
```

| # | 어디서 | 무엇을 함 |
|---|---|---|
| 1 | 브라우저 | 이메일 형식·비번 8자 검증 — 실패 시 인라인 에러, 흐름 중단 |
| 2 | 브라우저 | 상태 `submitting` — 버튼 회전, 입력란 비활성 |
| 3 | 브라우저 | `supabase.auth.signUp({ email, password })` 호출 |
| 4 | Supabase Auth | 비밀번호 bcrypt 해싱 |
| 5 | Supabase DB | `auth.users`에 새 row INSERT |
| 6 | Supabase Auth | 세션 토큰(JWT) 발급 |
| 7 | Supabase Auth | 응답 반환 (`user`, `session`) |
| 8 | 브라우저 | `@supabase/ssr`이 세션 쿠키 자동 설정 |
| 9 | 브라우저 | `router.push('/')` → 대시보드 → 상태 `success` |

**실패 분기**
| 케이스 | Supabase 응답 | 처리 |
|---|---|---|
| 이미 가입된 이메일 | `error.code = 'user_already_exists'` | 폼 상단 박스: "이미 가입된 이메일입니다." + "로그인하기" 링크 |
| 네트워크 오류 | fetch reject | 폼 상단 박스: "잠시 후 다시 시도해주세요." |

### 3.2 흐름 2 — 구글 OAuth 가입/로그인

**시나리오**: `/signup` 또는 `/login`에서 "구글로 계속하기" 클릭

```
[사용자]   [브라우저]                  [Supabase]            [구글]          [Supabase DB]
   │ 클릭    │ 1. signInWithOAuth({     │                    │                  │
   │        │      provider: 'google', │                    │                  │
   │        │      redirectTo:         │                    │                  │
   │        │      '/auth/callback' }) │                    │                  │
   │        ├────────────────────────►│                    │                  │
   │        │◄── 2. 구글 OAuth URL ─────┤                    │                  │
   │        │ 3. 구글 페이지로 이동       │                    │                  │
   │        ├────────────────────────────────────────────►│                  │
   │ 동의   ├────────────────────────────────────────────►│                  │
   │        │                          │◄── 4. callback ───┤                  │
   │        │                          │ 5. code↔token 교환 │                  │
   │        │                          │ 6. 신규면 INSERT ────────────────────►│
   │        │◄── 7. /auth/callback ────┤                    │                  │
   │        │ 8. 세션 확정, '/' redirect │                    │                  │
   │ 대시보드│◄──                        │                    │                  │
```

| # | 어디서 | 무엇을 함 |
|---|---|---|
| 1 | 브라우저 | `supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo: '<origin>/auth/callback' } })` 호출 |
| 2 | Supabase | 구글 OAuth 인증 URL 생성·반환 |
| 3 | 브라우저 | 구글 로그인/동의 페이지로 이동 |
| 4 | 구글 → Supabase | 사용자 동의 후 Supabase callback URL로 redirect (인가 코드 포함) |
| 5 | Supabase | 인가 코드 ↔ 토큰 교환, 구글 프로필 정보 수신 |
| 6 | Supabase DB | 신규 사용자면 `auth.users`에 INSERT (`raw_app_meta_data.provider = 'google'`) |
| 7 | Supabase | 우리 사이트의 `/auth/callback`로 redirect (세션 정보 포함) |
| 8 | 우리 `/auth/callback` 라우트 핸들러 | 세션 쿠키 확정 후 `/`로 redirect |

**실패 분기**
| 케이스 | 처리 |
|---|---|
| 사용자가 구글 동의 거부 | Supabase callback에 error 파라미터 → `/login`으로 redirect + 폼 상단 "구글 로그인을 완료하지 못했습니다." |
| 네트워크/타임아웃 | `/login`으로 redirect + "잠시 후 다시 시도해주세요." |

### 3.5 흐름 5 — 비밀번호 재설정 요청

**시나리오**: `/reset-password`에서 이메일 입력 → "재설정 메일 보내기" 클릭

| # | 어디서 | 무엇을 함 |
|---|---|---|
| 1 | 브라우저 | 이메일 형식 검증 (실패 시 인라인 에러) |

| 5 | 브라우저 | 상태 `sent` → 카드 내용을 "메일을 확인해주세요" 화면으로 교체 |

### 3.6 흐름 6 — 새 비밀번호 설정

**시나리오**: 메일의 재설정 링크 클릭 → `/reset-password/new`에서 새 비밀번호 입력

| # | 어디서 | 무엇을 함 |
|---|---|---|
| 1 | 사용자 | 메일의 재설정 링크 클릭 |
| 2 | 브라우저 | Supabase callback URL → 우리 `/auth/callback?next=/reset-password/new` 도착 |
| 3 | 우리 `/auth/callback` 라우트 | 토큰을 recovery 세션으로 교환 |
| 4 | 브라우저 | `/reset-password/new`로 redirect |
| 5 | `/reset-password/new` | 진입 시 recovery 세션 확인 — 없거나 만료면 `invalid-token` 상태 → 만료 화면 |
| 6 | 사용자 | 새 비밀번호 입력 (실시간 8자 검증) → "비밀번호 변경" 클릭 |
| 7 | 브라우저 | 상태 `submitting` → `supabase.auth.updateUser({ password: newPassword })` 호출 |
| 8 | Supabase | 비밀번호 해시 갱신, recovery 세션을 정상 세션으로 승격 |
| 9 | 브라우저 | 상태 `success` → `router.push('/')` |

**실패 분기**
| 케이스 | 처리 |
|---|---|
| 토큰 만료/무효 | 진입 시점에 invalid-token 화면 표시 ("링크가 만료되었습니다" + "다시 요청하기" 버튼) |
| 네트워크 오류 | 폼 상단 박스: "잠시 후 다시 시도해주세요." |

### 3.7 흐름 7 — 인증 미들웨어 (라우트 보호)

**시나리오**: 모든 페이지 요청에 대해 백그라운드에서 자동 실행 (사용자가 명시적으로 트리거하지 않음)

| # | 어디서 | 무엇을 함 |
|---|---|---|
| 1 | 브라우저 → 서버 | 사용자가 임의 경로(`/`, `/signup`, `/login`, …)에 요청 |
| 2 | `middleware.ts` (Next.js) | `@supabase/ssr`로 요청 쿠키에서 세션 확인 |
| 3-A | 로그인 상태인데 비로그인 전용 경로(`/login`, `/signup`, `/reset-password*`)에 요청 | → `/`로 redirect |
| 3-B | 비로그인 상태인데 보호 경로(`/` 및 향후 추가 경로)에 요청 | → `/login`으로 redirect |
| 3-C | 그 외 정상 | 통과 |
| 4 | Next.js | 결정된 경로로 라우팅 |

**보호 대상 정의**: `middleware.ts`의 `config.matcher`에서 관리.
- 공개 경로: `/login`, `/signup`, `/reset-password`, `/reset-password/new`, `/auth/callback`
- 보호 경로: 그 외 전부 (`/`, `/mypage` 포함)

### 3.8 흐름 8 — 닉네임 추가/수정

**시나리오**: `/mypage`에서 닉네임 영역 "추가"/"수정" → 입력 → "저장"

| # | 어디서 | 무엇을 함 |
|---|---|---|
| 1 | 브라우저 | "추가"/"수정" 클릭 → 닉네임 영역만 editing 상태로 전환 |
| 2 | 사용자 | 새 닉네임 입력 → "저장" |
| 3 | 브라우저 | 클라이언트 검증 (trim 후 1~20자) → 통과 시 서버 액션 호출 |
| 4 | 서버 액션 (`updateNickname`) | `getUser()` → 본인 식별, 길이 재검증 |
| 5 | Supabase DB | `UPDATE public.profiles SET nickname=$1 WHERE id=auth.uid()` |
| 6 | DB | 중복이면 UNIQUE 위반 (PostgreSQL error 23505) |
| 7 | 서버 액션 | 성공: `revalidatePath('/mypage')` → 반환 |
| 8 | 브라우저 | idle 상태 복귀, 새 닉네임 즉시 표시 |

**실패 분기**
| 케이스 | 처리 |
|---|---|
| 빈 값(trim 후) | 인라인 에러 `닉네임을 입력해주세요` |
| 21자 이상 | 인라인 에러 `닉네임은 20자 이하여야 합니다` |
| 중복 (UNIQUE 위반) | 인라인 에러 `이미 사용 중인 닉네임입니다` |
| 네트워크/기타 | 카드 상단 박스 `잠시 후 다시 시도해주세요.` |
